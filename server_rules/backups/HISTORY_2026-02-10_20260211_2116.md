## Access Control v1.1 — Detailed Spec Notes (User-Provided Logic)

### 목적/원칙 요약
- 초대한 사용자만 APK 설치/접속 가능.
- 비밀번호/토큰 평문 저장 금지 (서버는 해시만).
- force_password_change=true 시 메인 기능 접근 금지.
- 비번 변경/리셋/차단 시 기존 세션 전부 무효화.
- 생체 인증은 “서버 인증 대체”가 아니라 “토큰 Unlock” 용도.

### 초대(Invitation)
- 모드 2종(AUTO/MANUAL) 동시 지원.
- AUTO: 서버가 강한 랜덤 initial_password 발급, 1회만 노출.
- MANUAL: 관리자 지정 가능(전화번호 등), 레이트리밋/잠금/만료 방어장치 필수.
- 저장 항목: user_code, role, status, password_hash, force_password_change=true, invite_status, expires_at, memo, created_at, updated_at, device_binding_enabled.
- invite_status 전이: CREATED → SENT → ACTIVATED → PASSWORD_CHANGED → ACTIVE.

### 최초 접속 플로우
- 입력: user_code + initial_password.
- 검증: status/blocked/locked/expires_at/디바이스 바인딩 체크.
- 성공 시 토큰 발급(임시), force_password_change 유지.
- 프로필(name/phone) 1회 등록 후 invite_status=ACTIVATED.
- 비밀번호 변경 강제 완료 후에만 메인 접근 허용.
- 비번 변경 완료 시 기존 세션 revoke + 신규 토큰 발급.

### 일반 로그인/세션
- 입력: user_code + password.
- 서버 DB에 token_hash 저장, 원문 금지.
- logout: 현재 세션 revoke.
- MASTER는 유저 전체 세션 revoke 가능.

### 생체 인증
- ON: 토큰을 단말 보안 저장소에 저장 → 바이오 인증으로 해제.
- OFF: 자동 로그인 불가(토큰 잠금/삭제).

### MASTER 기능
- 유저 리스트/상세/차단/해제/비번 리셋/만료 연장/권한 변경/디바이스 바인딩 제어.
- 최소 1 MASTER 보호 정책.
- 모든 관리자 액션 감사로그 기록.

### 로그/감사
- login_events: 모든 로그인 시도 기록 (success/fail + reason_code).
- admin_audit_logs: 모든 관리자 조작 기록.

### 보안 방어장치
- user_code 기준 실패횟수에 따른 잠금(locked_until).
- IP 기준 레이트리밋.
- 임시비번 만료(expires_at) 필수.

### 사용자 메시지
- blocked/locked/expired/force change 상황별 안내 메시지 제공.

### 테스트 시나리오 (요구사항)
- AUTO 초대 → 최초 로그인 → 프로필 → 비번 변경 → 메인 접근.
- MANUAL 초대(전화번호) → 최초 로그인 → 비번 변경.
- 임시비번 만료 → 로그인 실패.

## Summary (미장 미노출 장애 원인분석 + 회고 + 재발방지 반영)
- 사용자 이슈: 미장 탭에서 종목이 "없음/빈 화면"으로 보임.

## Root Cause (증거 기반)
1. 앱 네트워크 읽기 타임아웃이 20초였음.
   - 파일: `app/src/main/java/com/example/stock/data/api/NetworkModule.kt`
   - 기존 설정: `readTimeout(20s)`
2. 미장 API는 Form4 원문 스캔 구조라 20초를 자주 초과.
   - 실측: `max_candidates=80` 요청도 `time_total=21.445s` (200 응답)
   - 즉, 기본 20초로는 정상 응답 직전 끊길 수 있는 구조.
3. 직전 변경에서 미장 요청량(`max_candidates`)을 220까지 올려 응답시간이 더 길어졌고,
   이로 인해 타임아웃/취소 빈도가 증가.
4. 운영 access log에서 동일 시점 증거 확인:
   - `/market/us-insiders?...max_candidates=220` 다수 `499`
   - 동일 조건을 curl로 호출하면 최종 `200` + `returned_count=1` 정상 데이터 확인.

## Non-Root / 혼동 요인
- KRX/FDR 로그 에러는 premarket/benchmark 경로의 노이즈로, 미장 Form4 파이프라인 직접 원인은 아님.
- 미장 자체 데이터는 존재함(IRIX 1건 확인).

## Fixes Applied
- 앱 타임아웃 분리:
  - `NetworkModule`에 `slowApi()` 추가(read timeout 90s)
  - 미장 호출은 `slowApi()` 사용하도록 분리
- 미장 요청량 조정(속도/안정성 밸런스):
  - 앱 기본 `max_candidates`를 80으로 조정

     모바일 클라이언트 타임아웃 예산(20초)과의 충돌을 사전에 검증하지 못함.
